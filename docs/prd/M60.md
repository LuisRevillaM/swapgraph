# M60 — Rollout freeze export overlay + control observability refinements (fixtures-first)

## Objective
Harden operator governance by adding an optional freeze-window export overlay and exposing richer rollout-policy observability on the partner self-serve read surface.

## Scope

### 1) Freeze-window export overlay
Add an operator gate for settlement vault reconciliation exports:
- `PARTNER_PROGRAM_ROLLOUT_POLICY_FREEZE_EXPORT_ENFORCE=1`
- when enabled and rollout policy `controls.freeze_active=true`, block exports with:
  - `FORBIDDEN`
  - `reason_code=partner_rollout_frozen`

When disabled (default), freeze windows continue to block policy mutation only (M59 behavior), not export reads.

### 2) Partner rollout observability expansion
Extend `partnerProgram.vault_export.get` response `rollout_policy` projection with:
- `freeze_export_enforced`
- `policy_source`
- `policy_version`
- `policy_updated_at`
- `policy_updated_by`
- `last_admin_action_at`
- `last_admin_action_by`

Also include `partner_rollout_frozen` in `reasons[]` when freeze export overlay is enabled and freeze is active.

### 3) Contract/docs updates
Update:
- API behavior docs for freeze overlay semantics.
- Error docs to document freeze overlay reason-code usage.
- `PartnerProgramVaultExportGetResponse` schema + example.

### 4) Fixtures-first verifier
Deterministic scenario proves:
- freeze active + overlay disabled ⇒ export still allowed,
- overlay enabled ⇒ partner status shows frozen reason and export fails with `partner_rollout_frozen`,
- clearing controls restores export path,
- signed export verify + tamper-fail checks remain intact,
- observability fields report policy source/version/update/admin-action metadata deterministically.
